% ABSTRACT
% ============================================================================
\begin{abstract}
NCCL is the de facto standard library for GPU collective communication
in large-scale distributed training. Its plugin system allows
operators to customize collective behavior, but plugins execute as
unverified native code: a null-pointer dereference crashes the
training job; plugins share no state across extension points; and
updating a plugin requires restarting the job.
We observe that NCCL's CPU-side decision path is latency-tolerant:
the GPU collective it configures takes microseconds to milliseconds, making
eBPF's verification overhead negligible in exchange for safety,
composability, and live update.
We present \sysname, which embeds a userspace eBPF runtime inside
NCCL's plugin interface, requiring zero NCCL modifications,
to provide (1)~load-time verification that catches crash-inducing
bugs before execution, (2)~typed eBPF maps for cross-plugin state
sharing, and (3)~atomic policy hot-reload without call loss.
Our evaluation on NCCL~2.29.7 shows 52--100\ns of CPU overhead
per decision ($<$0.06\% of collective latency), correct rejection
of all unsafe programs, sub-microsecond policy swaps, and
elimination of intermittent 2$\times$ AllGather throughput
degradation caused by non-deterministic default behavior, with
zero measurable GPU-level overhead.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Large-scale GPU training depends on collective communication
performance. Frameworks such as Megatron-LM~\cite{megatron},
DeepSpeed~\cite{deepspeed}, and PyTorch FSDP~\cite{pytorch-fsdp}
rely on NCCL~\cite{nccl} for all inter-GPU data movement.
The choice of algorithm (ring vs.\ tree), protocol (LL vs.\ Simple),
and channel count significantly affects collective
latency~\cite{autoccl, msccl}. NCCL exposes a plugin
system (tuner, profiler, net, and env plugins) that allows operators
to customize these decisions, and major cloud providers already ship
custom tuner plugins~\cite{autoccl}.

However, every plugin executes as native code in NCCL's address space.
This creates three problems:

\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{No verification.}
A null-pointer dereference crashes the training job; an infinite loop
hangs the communicator; an off-by-one error silently corrupts NCCL
internal state. In clusters where jobs run for days across thousands of
GPUs, such failures waste enormous resources and are difficult to
diagnose.

\item \textbf{No cross-plugin state sharing.}
NCCL's tuner and profiler are independent extension points with no
communication mechanism. A profiler that measures collective latency
cannot pass that information to a tuner that could adapt, precluding
closed-loop control.

\item \textbf{No live update.}
Updating a plugin requires restarting the job. Cluster operators that
must iterate on tuning policies across heterogeneous clusters have no
way to roll out changes safely at runtime.
\end{itemize}

We observe that eBPF's execution model is well-suited to
this setting. The key insight is that NCCL's policy hot path
(\texttt{getCollInfo}) runs on the CPU and is
\emph{latency-tolerant}: the GPU collective it configures
takes microseconds to milliseconds, so 50--100\ns of eBPF
overhead is negligible. This contrasts with eBPF's traditional
domain (packet processing and syscall filtering), where
nanosecond budgets are tight. In collective communication,
the overhead budget is generous, and eBPF's capabilities map
directly to the gaps above: load-time verification catches
memory-safety and termination bugs, typed maps enable cross-plugin state
sharing, and atomic program replacement provides hot-reload.

\sysname embeds bpftime~\cite{bpftime}, a userspace eBPF runtime,
inside a standard NCCL plugin. Policy authors write restricted~C
compiled to BPF bytecode, verified at load time, and JIT-compiled to
native code. The system loads via standard
\texttt{NCCL\_TUNER\_PLUGIN} and \texttt{NCCL\_PROFILER\_PLUGIN}
environment variables, requiring zero modifications to NCCL.
This paper makes four contributions:

\begin{enumerate}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{\sysname}, a system that embeds verified eBPF policy
execution inside NCCL's plugin interface with no NCCL modifications
(\S\ref{sec:design}--\ref{sec:impl}).

\item A map-based composability model that enables closed-loop
profiler-to-tuner adaptation, a capability absent from NCCL's
native plugin architecture (\S\ref{sec:design}).

\item Atomic policy hot-reload with 0.774\us swap time and zero
call loss across 400{,}000 invocations (\S\ref{sec:design}).

\item An evaluation on NCCL~2.29.7 showing 52--100\ns CPU overhead
per decision ($<$0.06\% of collective latency), end-to-end GPU
overhead within measurement noise, and elimination of
intermittent 2$\times$ AllGather throughput degradation
(\S\ref{sec:eval}).
\end{enumerate}

\section{Background and Motivation}
\label{sec:motivation}

\paragraph{NCCL plugin architecture.}
NCCL organizes extensibility around four plugin types loaded via
\texttt{dlopen}: the \emph{tuner} plugin's \texttt{getCollInfo}
selects algorithm, protocol, and channel count per collective call;
the \emph{profiler} plugin receives timestamped event callbacks; the
\emph{net} plugin controls network connections; and the \emph{env}
plugin sets runtime parameters. Major training frameworks (Megatron-LM~\cite{megatron}, DeepSpeed~\cite{deepspeed},
PyTorch FSDP~\cite{pytorch-fsdp}, and JAX~\cite{pjrt}) all
depend on NCCL's collective primitives. These plugins are
independent extension points with no cross-plugin communication
mechanism and no verification of correctness.

\paragraph{The safety gap.}
Native plugins execute with full process privileges. A null
dereference produces an immediate \texttt{SIGSEGV}, terminating the
training job. An off-by-one error can silently corrupt NCCL internal
state, producing subtly incorrect algorithm selections that degrade
training convergence without any crash or error message. This failure
mode is worse than a crash because it is silent. An infinite loop hangs
the communicator indefinitely. In all cases, the root cause is that
\texttt{dlopen}-based extensions have no safety net. The problem
is compounded when operators must roll out plugin updates across a fleet
without a mechanism for safe hot-reload.

\paragraph{Why eBPF fits.}
Three properties make eBPF well-suited: (1)~a generous overhead
budget (\texttt{getCollInfo()} runs on the CPU before GPU-side
collectives that take microseconds to milliseconds, so 50--100\ns of
eBPF overhead is negligible); (2)~typed maps that provide concurrent,
verified state sharing between profiler and tuner without ad-hoc shared
memory; and (3)~atomic program replacement for hot-reload with
verification of the replacement before swap.

% 3. DESIGN

\section{Design}
\label{sec:design}

\subsection{Design Tensions}

\paragraph{T1: Safety vs.\ overhead.}
Verification at every invocation would add milliseconds; verification
only at load time (1--5\,ms, amortized over the job lifetime) trades
per-call checks for a one-time cost. This relies on the invariant that
verified BPF bytecode, once JIT-compiled, cannot violate its safety
guarantees at runtime. The tradeoff: verification catches memory
safety and termination bugs, not semantic correctness.

\paragraph{T2: Structured sharing vs.\ flexibility.}
All cross-component state sharing uses typed eBPF maps with atomic
access semantics. Maps constrain policies to fixed-size keys/values,
but NCCL policy state is inherently simple (per-communicator scalars),
and structured sharing eliminates the locking bugs that plague ad-hoc
native plugin state sharing.

\paragraph{T3: Availability vs.\ consistency during updates.}
Hot-reload atomically swaps the function pointer; any in-progress call
completes under the old policy, the next call uses the new one. No
call is lost. If the new policy fails verification, the old policy
continues. The tradeoff: different threads may briefly execute
different policies, which is acceptable because collective decisions
are independent per call.

\paragraph{T4: Compatibility vs.\ capability.}
\sysname operates within NCCL's existing plugin ABI, requiring no source
modifications, no custom builds, and no kernel modules. The action space
is limited to algorithm, protocol, and channel count selection.
Deeper interventions are outside scope, but the exposed knobs cover
the most impactful tuning decisions. \sysname selects among
NCCL's built-in algorithms; it does not implement custom
collective algorithms.

\subsection{Threat Model}
\label{sec:threat-model}

Policies are written by platform operators, not untrusted tenants.
The eBPF verifier (PREVAIL-based) checks memory safety, bounded
execution, stack safety, and helper whitelisting. The system does
\emph{not} protect against resource exhaustion, side channels, bugs in
the trusted computing base (bpftime JIT, PREVAIL verifier, host
plugin), or semantically incorrect decisions. This provides defense in
depth (eliminating crashes, hangs, and memory corruption), analogous
to the Linux kernel's eBPF model.

\subsection{Architecture}

\begin{figure*}[t]
  \centering
  \input{figures/architecture}
  \caption{\sysname architecture. Policy programs are verified via
  PREVAIL, JIT-compiled by bpftime, and execute inside the NCCL
  process. Profiler and tuner communicate through shared typed eBPF
  maps. The entire system loads as standard NCCL plugins.}
  \Description{Architecture diagram showing NCCL Core, Tuner and
  Profiler adapters, bpftime runtime with verifier/JIT/maps/helpers,
  and eBPF policy programs communicating through shared maps.}
  \label{fig:architecture}
\end{figure*}

Figure~\ref{fig:architecture} shows the architecture. Policy authors
write restricted C compiled to BPF ELF objects. At load time, each
program is verified (abstract interpretation over all execution paths)
and JIT-compiled to x86-64 code. The map subsystem provides
intra-policy state and cross-plugin communication: a profiler eBPF
program writes latency observations; the tuner reads them. Hot-reload
verifies and JIT-compiles a replacement, then atomically swaps the
function pointer via compare-and-swap.

\paragraph{Policy programming model.}
A policy takes a \texttt{policy\_context} (collective type, message
size, rank count) and writes algorithm, protocol, and channel count
decisions. The verifier ensures policies only read input fields and
write output fields. Map lookups require null checks before
dereference, the same check that prevents the most common native
plugin crash (null-pointer dereference after failed lookup).

\begin{lstlisting}[language=C, caption={Size-aware tuning policy
  (8 lines of logic). Messages $\leq$32\,KB use TREE/Simple
  (lower latency); larger messages use RING/Simple
  (higher throughput). Always selecting Simple eliminates
  the AllGather variance in Table~\ref{tab:stability}.}, label={lst:size-aware},
  basicstyle=\ttfamily\scriptsize]
SEC("policy")
int size_aware(struct policy_context *ctx) {
  if (ctx->msg_size <= 32 * 1024) {
    ctx->algorithm = NCCL_ALGO_TREE;
    ctx->protocol  = NCCL_PROTO_SIMPLE;
  } else {
    ctx->algorithm = NCCL_ALGO_RING;
    ctx->protocol  = NCCL_PROTO_SIMPLE;
  }
  return 0;
}
\end{lstlisting}

More complex policies read profiler-provided latency from eBPF
maps and adapt channel count dynamically:

\begin{lstlisting}[language=C, caption={Profiler-informed channel
  adaptation. The \texttt{latency\_map} is written by the profiler
  plugin and read by the tuner policy.}, label={lst:adaptive},
  basicstyle=\ttfamily\scriptsize]
SEC("policy")
int adaptive_channels(struct policy_context *ctx) {
  __u32 key = ctx->comm_id;
  struct latency_state *st =
    bpf_map_lookup_elem(&latency_map, &key);
  if (!st) { /* first call: initialize */
    ctx->n_channels = 8; return 0;
  }
  if (st->avg_latency_ns > 1000000)
    st->channels = min(st->channels + 1, 16);
  ctx->n_channels = st->channels;
  return 0;
}
\end{lstlisting}

The mandatory null check on line~5 is enforced by the verifier;
omitting it would cause load-time rejection.
This closed-loop control has no built-in support in NCCL's native
plugin architecture, where tuner and profiler plugins share no state
mechanism.

\section{Implementation}
\label{sec:impl}

\sysname is implemented in approximately 2{,}500 lines of C/C++
(plugin host) and 500 lines of restricted C (policy programs).

\paragraph{bpftime integration.}
We embed bpftime~\cite{bpftime} as the userspace eBPF runtime.
bpftime provides an LLVM-based JIT compiler, a
PREVAIL-based~\cite{prevail} static verifier, and a map subsystem.
The verifier runs in the same address space as the plugin, adding
approximately 1--5\,ms of one-time startup overhead amortized over
the lifetime of the training job. The LLVM JIT produces optimized
x86-64 code that benefits from the same optimization passes as
\texttt{clang -O2}, reducing the gap between JIT-compiled eBPF and
native code.

\paragraph{Plugin registration.}
\sysname registers itself as both a tuner and profiler plugin via
NCCL's \texttt{ncclTunerPlugin\_v3} (or v5) and
\texttt{ncclProfilerPlugin\_v1} (or v6) interfaces. The tuner
plugin's \texttt{getCollInfo()} function constructs a
\texttt{policy\_context}, invokes the JIT-compiled eBPF function,
and copies the output fields into NCCL's result structure. The
profiler plugin's event callbacks extract latency information and
update the shared eBPF maps.

\paragraph{NCCL integration challenges.}
NCCL's tuner API uses cost arrays rather than direct algorithm IDs:
the tuner sets costs to zero for preferred choices and to a sentinel
value (\texttt{1e9}) for others, allowing NCCL to fall back
gracefully if the requested combination is unavailable. Our
integration translates eBPF policy outputs into cost array entries.
NCCL also passes a maximum channel count that the tuner must respect;
our native baseline layer clamps the policy's request. Communicator
identification required deriving a stable ID from the context pointer
via hashing, since the communicator ID is not directly exposed in the
tuner API as a simple integer.

\paragraph{Hot-reload mechanism.}
The active policy is stored as an atomic function pointer. Reload has
three phases: (1)~verify the new BPF program, (2)~JIT-compile it,
(3)~atomic compare-and-swap on the pointer. The total reload cost is
$\sim$7.3\,ms; only the final swap (0.774\us) touches the hot path.
The old pointer is retained until in-flight calls drain. If the
replacement fails verification, the swap is aborted and the old policy
continues; the system never enters an unverified state.

\paragraph{Native baseline for comparison.}
To provide a fair overhead comparison, we implemented a native C++
baseline that performs identical policy logic without the eBPF layer,
compiled with the same optimization flags (\texttt{-O2}). The
overhead difference between the native baseline and eBPF policies
directly measures the cost of the eBPF dispatch and JIT execution
layers, isolated from any policy logic cost.

\section{Evaluation}
\label{sec:eval}

\paragraph{Testbed.}
CPU microbenchmarks run on a 24-core x86-64 platform with isolated
cores; each benchmark performs 1\,million calls and reports P50/P99
latencies. GPU experiments use a single NVIDIA RTX~5090 with
CUDA~12.9 and NCCL~2.29.7, running 2~MPI ranks on the same host
over socket transport.

tested configurations (no plugin, \texttt{noop},
\texttt{size\_aware\_v2}, \texttt{slo\_enforcer}) produce statistically
indistinguishable AllReduce latencies.

% --------------------------------------------------------------------------
\subsection{Safety and Hot-Reload}
\label{sec:eval-verifier}

We tested 14 eBPF programs against the PREVAIL-based verifier: 7~safe
policies (including all policies in Table~\ref{tab:overhead}) were
accepted; 7~unsafe programs, each targeting a distinct bug class
(null-pointer dereference, out-of-bounds access, illegal helper,
stack overflow, unbounded loop, input-field write, division by
zero), were rejected at load time with actionable error messages.

The safety difference is concrete. We implemented the same
null-dereference bug as both a native C plugin and an eBPF policy:

\begin{lstlisting}[basicstyle=\ttfamily\scriptsize,numbers=none,frame=none,xleftmargin=1em]
Native plugin:  Signal: SIGSEGV (address 0x0)
                  in getCollInfo() at native_bad_plugin.so
eBPF policy:    VERIFIER REJECT: R0 is a pointer to
                  map_value_or_null; must check != NULL
                  before dereference at insn 7
\end{lstlisting}

\noindent The native plugin crashes the training job; the eBPF policy
is caught before execution.

Hot-reload swaps the active policy atomically via compare-and-swap
(0.774\us swap time; $\sim$7.3\,ms including verification and JIT).
Across 400{,}000 continuous invocations, we observe zero lost calls.
If the replacement fails verification, the old policy continues
uninterrupted.

NCCL's default tuner selects algorithm and protocol based on a
static cost model computed at initialization. We observe that
AllGather under the default configuration exhibits
\emph{high run-to-run variance}: across 3~independent
runs at 128\,MiB, 2~runs produced only 2.08--2.11\,GB/s while
the third achieved 4.61\,GB/s (Table~\ref{tab:stability}).
The slow runs show degraded throughput at \emph{all} message
sizes (even 1\,MiB: 0.23 vs.\ 0.48\,GB/s). We hypothesize that
NCCL's topology-dependent initialization intermittently assigns
a suboptimal internal configuration.

Forcing \texttt{NCCL\_PROTO=Simple} eliminates this variance
entirely: all 3~runs produce 4.62$\pm$0.01\,GB/s at 128\,MiB.
When DEFAULT operates normally (run~3), its throughput matches
Simple (4.61 vs.\ 4.63\,GB/s), suggesting that the degradation
is not inherent to the protocol choice but to non-deterministic
initialization.

Our \texttt{size\_aware} policy (Listing~\ref{lst:size-aware})
always selects Simple protocol, which as a side effect eliminates
this run-to-run variance. This demonstrates a policy value beyond
throughput optimization: \emph{improved stability} through
consistent protocol steering. The policy adds zero measurable
GPU-level overhead versus NCCL's default when DEFAULT operates
in its fast mode.

Beyond protocol selection, the eBPF programming model supports
closed-loop profiler-to-tuner adaptation via shared maps
and per-communicator differentiation for multi-tenant SLO
enforcement, capabilities absent from NCCL's native plugin
architecture.

\section{Related Work}
\label{sec:related}

\paragraph{Collective communication.}
MSCCL~\cite{msccl} and TACCL~\cite{taccl} synthesize static
schedules offline; they do not address runtime policy selection or
safety. AutoCCL~\cite{autoccl} automates tuning via search but uses
native code without verification. MSCCL++~\cite{mscclpp} provides
GPU-driven primitives at the mechanism layer.
Megatron-LM~\cite{megatron} and DeepSpeed~\cite{deepspeed} implement
custom collective strategies at the framework level but rely on
NCCL's native plugin interface without safety guarantees. \sysname
operates at the policy layer and is complementary: a~policy can
activate synthesized schedules based on dynamic conditions.

\paragraph{eBPF for systems extensibility.}
XRP~\cite{xrp} embeds eBPF in the kernel's NVMe path, exploiting
the same insight that certain system paths are latency-tolerant
enough for eBPF overhead. Electrode~\cite{electrode} accelerates
distributed protocols; eTran~\cite{etran} applies eBPF to RDMA
network functions; PageFlex~\cite{pageflex} extends eBPF to page
management. No prior work applies eBPF to GPU collective
communication. bpftime~\cite{bpftime} provides the userspace eBPF
runtime substrate that \sysname builds upon.

\paragraph{Alternative safe-extension mechanisms.}
WebAssembly provides sandboxed execution but lacks eBPF's
static termination and memory-safety verification; its runtime
overhead is typically higher due to bounds checks that eBPF
eliminates statically. Out-of-process services provide strong
isolation but add microseconds of IPC latency, which is an order of
magnitude more than \sysname's 52--100\ns in-process overhead.
Declarative DSLs cannot express stateful policies or feedback
loops. Hardware isolation (Intel MPK) provides memory separation
without verification. eBPF provides both verification and
efficient execution in a single model.

\section{Discussion and Conclusion}
\label{sec:conclusion}

\paragraph{Limitations and future work.}
\sysname currently targets NCCL's tuner and profiler interfaces; the
net plugin involves complex state (RDMA queue pairs, memory
registrations) that does not map cleanly to eBPF's restricted model.
Our evaluation uses 2~ranks on a single node; multi-node experiments
with InfiniBand transport and large rank counts are important future
work. The restricted C programming model is familiar to eBPF
developers but presents a barrier for ML engineers; a higher-level
policy DSL compiled to BPF would improve usability.

Not all policies can be expressed in eBPF: floating-point arithmetic,
dynamic memory allocation, and GPU memory access exceed the execution
model. These limitations are acceptable because policy decisions
operate on metadata (message size, rank count, latency), not tensor
content.

Our evaluation demonstrates that the mechanism works: a
size-aware eBPF policy mitigates intermittent 2$\times$
AllGather throughput degradation from non-deterministic default
behavior, with zero measurable GPU-level overhead. Full
evaluation on multi-node production workloads at scale is the
key next step.

\paragraph{Broader applicability.}
The eBPF-for-extensions pattern demonstrated by \sysname may be
applicable beyond NCCL. AMD's RCCL~\cite{rccl} shares the same plugin
architecture, making cross-vendor portability feasible. Other
libraries with native-code plugin interfaces (MPI implementations
such as OpenMPI and MPICH, distributed training
frameworks~\cite{pytorch-fsdp, deepspeed}, and RDMA middleware) face
similar safety gaps. The observation that GPU-adjacent CPU-side
decision paths have generous latency budgets for verified execution
may generalize to other parts of the accelerator software stack.

\paragraph{Conclusion.}
We presented \sysname, a system that transforms NCCL's unsafe native
plugin extension points into verified, composable policy hooks by
embedding a userspace eBPF runtime. The key insight is that NCCL's
CPU-side decision path is latency-tolerant enough to absorb eBPF
overhead. \sysname adds 52--100\ns per decision (under 0.06\% of
collective latency), catches common crash-inducing bugs at load time,
supports atomic hot-reload ($\sim$7.3\,ms total, 0.774\us on the
hot path) without call loss, and reduces AllGather throughput
variance from 50\% to 0.2\% coefficient of variation on our testbed. The system
requires zero
NCCL modifications and operates entirely within the stock plugin ABI.
