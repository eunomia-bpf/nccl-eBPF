% ============================================================================
% ABSTRACT
% ============================================================================
\begin{abstract}
NCCL's plugin system allows operators to customize collective
communication behavior, but plugins execute as native code with no
isolation, verification, or structured state sharing.
We present \sysname, which embeds a userspace eBPF runtime inside
NCCL's plugin interface to provide load-time verification,
typed eBPF maps for cross-plugin state sharing, and atomic policy
hot-reload. \sysname requires zero NCCL modifications.
Our evaluation shows 52--100\ns of CPU overhead per policy decision
($<$0.06\% of collective latency), 14/14 correct verifier verdicts
across seven safe and seven unsafe programs, and sub-microsecond
atomic policy swaps with zero call loss.
\end{abstract}

% 1. INTRODUCTION
% ============================================================================
\section{Introduction}
\label{sec:intro}

Large-scale GPU training depends on collective communication
performance. The choice of algorithm (ring vs.\ tree), protocol (LL
vs.\ Simple), and channel count can shift collective latency by orders
of magnitude~\cite{nccl, autoccl, msccl}. NVIDIA's NCCL~\cite{nccl}
exposes a plugin system---tuner, profiler, net, and env plugins---that
allows operators to customize these decisions. This extensibility has
enabled custom tuning at major cloud providers~\cite{autoccl}.

However, every NCCL plugin executes as native code in NCCL's address
space with no safety guarantees. A null-pointer dereference crashes
the training job; an infinite loop hangs the communicator; an
off-by-one error silently corrupts NCCL internal state. In production
clusters where jobs run for days across thousands of GPUs, such
failures waste enormous resources and are difficult to diagnose.
Furthermore, NCCL provides no mechanism for cross-plugin
communication---a profiler cannot pass latency measurements to a
tuner---and no way to update a plugin without restarting the job.

We observe that eBPF's execution model is a natural fit for NCCL
plugins. The key insight is that NCCL's policy hot path,
\texttt{getCollInfo()}, executes on the CPU and is latency-tolerant:
the collective it configures takes microseconds to milliseconds on the
GPU, making 50--100\ns of CPU overhead negligible. eBPF's map
abstraction directly addresses the cross-plugin state-sharing gap,
and eBPF's atomic program replacement enables policy hot-reload.

\sysname embeds bpftime~\cite{bpftime} inside a standard NCCL tuner
plugin. Policy authors write restricted C compiled to BPF bytecode,
verified at load time, and JIT-compiled to native code. The system
operates within NCCL's stock plugin ABI, requiring zero modifications
to NCCL itself. This paper makes four contributions:

\begin{enumerate}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item The case for verified extensibility in collective communication:
eBPF resolves the safety--customizability tension with negligible
overhead (\S\ref{sec:motivation}).

\item A map-based composability model enabling closed-loop
profiler-to-tuner adaptation (\S\ref{sec:design}).

\item Atomic policy hot-reload with 0.774\us swap time and zero
call loss across 400{,}000 invocations (\S\ref{sec:design}).

\item Evaluation on NCCL~2.29.7 with RTX~5090 hardware showing
52--100\ns CPU overhead and end-to-end GPU overhead within
measurement noise (\S\ref{sec:eval}).
\end{enumerate}

% 2. BACKGROUND AND MOTIVATION
% ============================================================================
\section{Background and Motivation}
\label{sec:motivation}

\paragraph{NCCL plugin architecture.}
NCCL organizes extensibility around four plugin types loaded via
\texttt{dlopen}: the \emph{tuner} plugin's \texttt{getCollInfo}
selects algorithm, protocol, and channel count per collective call;
the \emph{profiler} plugin receives timestamped event callbacks; the
\emph{net} plugin controls network connections; and the \emph{env}
plugin sets runtime parameters. These plugins are independent
extension points with no cross-plugin communication mechanism and
no verification of correctness.

\paragraph{The safety gap.}
Native plugins execute with full process privileges. A null
dereference produces an immediate \texttt{SIGSEGV}, terminating the
training job. An off-by-one error overwrites NCCL internal state,
causing silent corruption. An infinite loop hangs the communicator
indefinitely. In all cases, the root cause is that \texttt{dlopen}-based
extensions have no safety net. The situation worsens when operators
must roll out plugin updates across a fleet without a mechanism for
safe hot-reload.

\paragraph{Why eBPF fits.}
Three properties make eBPF well-suited: (1)~a generous overhead
budget---\texttt{getCollInfo()} runs on the CPU before GPU-side
collectives that take microseconds to milliseconds, so 50--100\ns of
eBPF overhead is negligible; (2)~typed maps that provide concurrent,
verified state sharing between profiler and tuner without ad-hoc shared
memory; and (3)~atomic program replacement for hot-reload with
verification of the replacement before swap.

% ============================================================================
% 3. DESIGN
% ============================================================================
\section{Design}
\label{sec:design}

\subsection{Design Tensions}

\paragraph{T1: Safety vs.\ overhead.}
Verification at every invocation would add milliseconds; verification
only at load time (1--5\,ms, amortized over the job lifetime) trades
per-call checks for a one-time cost. This relies on the invariant that
verified BPF bytecode, once JIT-compiled, cannot violate its safety
guarantees at runtime. The tradeoff: verification catches memory
safety and termination bugs, not semantic correctness.

\paragraph{T2: Structured sharing vs.\ flexibility.}
All cross-component state sharing uses typed eBPF maps with atomic
access semantics. Maps constrain policies to fixed-size keys/values,
but NCCL policy state is inherently simple (per-communicator scalars),
and structured sharing eliminates the locking bugs that plague ad-hoc
native plugin state sharing.

\paragraph{T3: Availability vs.\ consistency during updates.}
Hot-reload atomically swaps the function pointer; any in-progress call
completes under the old policy, the next call uses the new one. No
call is lost. If the new policy fails verification, the old policy
continues. The tradeoff: different threads may briefly execute
different policies, which is acceptable because collective decisions
are independent per call.

\paragraph{T4: Compatibility vs.\ capability.}
\sysname operates within NCCL's existing plugin ABI---no
source modifications, no custom builds, no kernel modules. The action
space is limited to algorithm, protocol, and channel count selection.
Deeper interventions are outside scope, but the exposed knobs cover
the most impactful tuning decisions.

\subsection{Threat Model}
\label{sec:threat-model}

Policies are written by platform operators, not untrusted tenants.
The eBPF verifier (PREVAIL-based) checks memory safety, bounded
execution, stack safety, and helper whitelisting. The system does
\emph{not} protect against resource exhaustion, side channels, bugs in
the trusted computing base (bpftime JIT, PREVAIL verifier, host
plugin), or semantically incorrect decisions. This provides defense in
depth---eliminating crashes, hangs, and memory corruption---analogous
to the Linux kernel's eBPF model.

\subsection{Architecture}

\begin{figure}[t]
  \centering
  \input{figures/architecture}
  \caption{\sysname architecture. Policy programs are verified via
  PREVAIL, JIT-compiled by bpftime, and execute inside the NCCL
  process. Profiler and tuner communicate through shared typed eBPF
  maps. The entire system loads as standard NCCL plugins.}
  \Description{Architecture diagram showing NCCL Core, Tuner and
  Profiler adapters, bpftime runtime with verifier/JIT/maps/helpers,
  and eBPF policy programs communicating through shared maps.}
  \label{fig:architecture}
\end{figure}

Figure~\ref{fig:architecture} shows the architecture. Policy authors
write restricted C compiled to BPF ELF objects. At load time, each
program is verified (abstract interpretation over all execution paths)
and JIT-compiled to x86-64 code. The map subsystem provides
intra-policy state and cross-plugin communication: a profiler eBPF
program writes latency observations; the tuner reads them. Hot-reload
verifies and JIT-compiles a replacement, then atomically swaps the
function pointer via compare-and-swap.

\paragraph{Policy programming model.}
A policy takes a \texttt{policy\_context} (collective type, message
size, rank count) and writes algorithm, protocol, and channel count
decisions. The verifier ensures policies only read input fields and
write output fields. Map lookups require null checks before
dereference---the same check that prevents the most common native
plugin crash (null-pointer dereference after failed lookup).

\begin{lstlisting}[language=C, caption={Size-aware tuning policy
  (12 lines of logic).}, label={lst:size-aware},
  basicstyle=\ttfamily\scriptsize]
SEC("policy")
int size_aware(struct policy_context *ctx) {
  if (ctx->msg_size <= 4096) {
    ctx->algorithm = NCCL_ALGO_TREE;
    ctx->protocol  = NCCL_PROTO_SIMPLE;
  } else if (ctx->msg_size <= 65536) {
    ctx->algorithm = NCCL_ALGO_TREE;
    ctx->protocol  = NCCL_PROTO_LL;
  } else {
    ctx->algorithm = NCCL_ALGO_RING;
    ctx->protocol  = NCCL_PROTO_LL;
  }
  return 0;
}
\end{lstlisting}

More complex policies read profiler-provided latency from eBPF
maps and adapt channel count dynamically. The adaptive channels
policy performs a map lookup per call, checks the result for null,
and adjusts channels based on observed latency. This closed-loop
control has no built-in support in NCCL's native plugin
architecture, where tuner and profiler plugins share no state
mechanism.

% 4. IMPLEMENTATION
% ============================================================================
\section{Implementation}
\label{sec:impl}

\sysname is implemented in approximately 2{,}500 lines of C/C++
(plugin host) and 500 lines of restricted C (policy programs).

\paragraph{NCCL integration challenges.}
NCCL's tuner API uses cost arrays rather than direct algorithm IDs:
the tuner sets costs to zero for preferred choices and to a sentinel
value for others. Our integration translates eBPF policy outputs into
cost array entries. NCCL also passes a maximum channel count that the
tuner must respect; our native baseline layer clamps the policy's
request. Communicator identification required deriving a stable ID
from the context pointer via hashing.

\paragraph{Hot-reload mechanism.}
The active policy is stored as an atomic function pointer. Reload has
three phases: (1)~verify the new BPF program, (2)~JIT-compile it,
(3)~atomic compare-and-swap on the pointer. The total reload cost is
$\sim$7.3\,ms; only the final swap (0.774\us) touches the hot path.
The old pointer is retained until in-flight calls drain. If the
replacement fails verification, the swap is aborted and the old policy
continues---the system never enters an unverified state.

\section{Evaluation}
\label{sec:eval}

We evaluate three research questions: (RQ1)~CPU overhead and
decomposition, (RQ2)~verifier effectiveness and hot-reload integrity,
and (RQ3)~policy expressiveness within the eBPF model.

\paragraph{Testbed.}
CPU microbenchmarks run on a 24-core x86-64 platform with isolated
cores; each benchmark performs 1\,million calls and reports P50/P99
latencies. GPU experiments use a single NVIDIA RTX~5090 with
CUDA~12.9 and NCCL~2.29.7, emulating 2~ranks over socket transport.
While modest in scale, this exercises the full plugin integration
path. The CPU-side policy overhead is independent of rank count.

approximately $51 + 26n_{\text{lookup}} + 25n_{\text{update}}$\ns.

P99 latencies track P50 within 20--25\ns, indicating minimal variance
from JIT-compiled code with no allocation or system calls.

\paragraph{Context.}
The most complex policy adds 100\ns over native code. For a 128\,MB
AllReduce (175\us on our GPU), this is 0.06\% of collective latency.
Even for 1\,KB messages ($\sim$2\us), the overhead is under 5\%.

\paragraph{End-to-end GPU overhead.}
Table~\ref{tab:e2e} shows GPU-measured collective latency at three
representative message sizes. The eBPF overhead is within measurement
noise across all sizes. The CPU-side 52--100\ns is invisible because
it occurs before the GPU collective launch, and NCCL's internal
processing between \texttt{getCollInfo()} and kernel launch adds
microseconds that dwarf the policy cost.

\paragraph{Limitations.}

transport. Production deployments with NUMA effects and concurrent GPU
DMA may show higher variance. The overhead formula is an empirical fit
to our policies; policies with larger map values may deviate. We do
not test multi-node or InfiniBand configurations.

Table~\ref{tab:verifier} shows verifier behavior on 14 programs.
Each unsafe program targets a specific failure mode: the same
null-dereference bug that crashes a native plugin with
\texttt{SIGSEGV} is rejected at load time with an actionable error
message (``must check != NULL before dereference''). The verifier
tracks register types, pointer provenance, and value ranges to detect
out-of-bounds access, unbounded loops, stack overflow, unauthorized
helpers, input-field writes, and potential division by zero.

\paragraph{Safety comparison.}
We implemented \texttt{bad\_lookup} as a native C plugin. Loading it
into NCCL crashes the process immediately. The eBPF version is
rejected before executing a single instruction. This is the core
safety proposition: common bugs that crash training jobs with native
plugins are caught at load time.

\paragraph{Hot-reload integrity.}
We continuously invoke \texttt{getCollInfo()} while swapping the
active policy. The atomic swap completes in 0.774\us; the full reload
(verify + JIT + swap) takes $\sim$7.3\,ms. Across 400{,}000
invocations, we observe zero failed or lost calls. The reload is
triggered at call 200{,}000; the first call under the new policy
occurs at call 254{,}709 (the intervening $\sim$54{,}700 calls
complete under the old policy while verification and JIT run
concurrently). When we attempt to hot-reload an unsafe policy, it is
rejected and the current policy continues uninterrupted. This
fail-safe property means operator mistakes during updates cannot crash
a running job.

\paragraph{Limitations.}
The verifier test suite is author-constructed; we do not claim
coverage of all possible unsafe patterns. A real-world evaluation
would benefit from a corpus of actual plugin bugs. The hot-reload test
uses a single process; we do not test across ranks or under real GPU
load.

% --------------------------------------------------------------------------
\subsection{RQ3: Policy Expressiveness}
\label{sec:eval-policies}

We demonstrate four policy capabilities beyond simple algorithm
selection. We frame this as an expressiveness demonstration: we show
that the eBPF programming model can express these patterns, not that
the resulting parameter changes improve end-to-end training outcomes.
Outcome evaluation on production workloads is future work.

\paragraph{Closed-loop adaptation.}
With the profiler plugin writing real collective latency into a shared
eBPF map, the \texttt{adaptive\_channels} tuner adjusts channels from
8 to 10 over successive collectives on our GPU testbed. Without
profiler feedback, the same policy remains static at 8 channels. This
confirms that profiler data flows through the eBPF map and affects
tuner decisions---a feedback loop with no built-in support in NCCL's
native plugin architecture.

\paragraph{Multi-tenant differentiation.}
The \texttt{slo\_enforcer} policy serves two communicators in the same
process with different SLOs: communicator~1 (SLO $\leq$ 1\,ms) receives
TREE / LL / 8~channels; communicator~2 (SLO $\leq$ 10\,ms) receives
RING / Simple / 1~channel. Per-communicator state in eBPF maps keyed
by \texttt{comm\_id} provides logical state separation.

\paragraph{Contention-aware adaptation.}
\begin{figure}[t]
  \centering
  \input{figures/contention_timeseries}
  \caption{Channel adaptation under changing contention. The policy
  increases channels during low latency and decreases under
  contention, with symmetric recovery. Contention is injected via
  elevated latency values in the profiler map.}
  \Description{Time-series showing channels rising to 12 in baseline
  phase, falling to 2 during contention, recovering to 12.}
  \label{fig:contention}
\end{figure}

Figure~\ref{fig:contention} shows the adaptive channels policy's
response to three contention phases over 300{,}000 calls. During
baseline (low latency), channels ramp to~12. Under injected
contention ($10\times$ latency), channels drop to~2. Upon recovery,
channels return to~12. This behavior emerges from the map-based
feedback loop without special contention-detection logic. We note
that contention was injected synthetically; stability under real
contention patterns remains to be validated.

\paragraph{Multi-collective coverage.}
\sysname policies work across AllReduce, AllGather, ReduceScatter,
and Broadcast. AllReduce selects TREE for small messages and RING for
large ones; the other collectives default to RING. A single policy
expresses collective-type-aware tuning strategies covering NCCL's full
range of collective operations.

\section{Related Work}
\label{sec:related}

\paragraph{Collective communication optimization.}
MSCCL~\cite{msccl} and TACCL~\cite{taccl} synthesize static
communication schedules offline; they do not address runtime policy
selection or safety. AutoCCL~\cite{autoccl} automates tuning via
search but uses native code without verification.
MSCCL++~\cite{mscclpp} provides GPU-driven communication primitives
at the mechanism layer. \sysname operates at the policy layer and is
complementary: a policy can activate synthesized schedules based on
dynamic conditions.

\paragraph{eBPF for systems extensibility.}
XRP~\cite{xrp} embeds eBPF in the kernel's NVMe path, exploiting
the same insight that certain system paths are latency-tolerant
enough for eBPF overhead. Electrode~\cite{electrode} accelerates
distributed protocols; eTran~\cite{etran} applies eBPF to RDMA
network functions; PageFlex~\cite{pageflex} extends eBPF to page
management. No prior work applies eBPF to GPU collective
communication. bpftime~\cite{bpftime} provides the userspace eBPF
runtime substrate that \sysname builds upon.

\paragraph{Alternative safe-extension mechanisms.}
WebAssembly provides sandboxed execution but lacks eBPF's static
termination and memory-safety verification; its runtime overhead is
typically higher due to bounds checks that eBPF eliminates statically.
Out-of-process services provide strong isolation but add microseconds
of IPC latency---an order of magnitude more than \sysname's
52--100\ns in-process overhead. Declarative DSLs are safe by
construction but cannot express stateful policies or feedback loops.
Hardware isolation (Intel MPK) provides memory separation without
verification; a policy can still loop indefinitely. eBPF provides
both verification and efficient execution in a single model with a
mature toolchain.

% 7. DISCUSSION AND CONCLUSION

\section{Discussion and Conclusion}
\label{sec:conclusion}

\paragraph{Limitations and future work.}
\sysname currently targets NCCL's tuner and profiler interfaces; the
net plugin involves complex state (RDMA queue pairs, memory
registrations) that does not map cleanly to eBPF's restricted model.
Our evaluation uses 2~ranks on a single node; multi-node experiments
with InfiniBand transport and large rank counts are important future
work. The restricted C programming model is familiar to eBPF
developers but presents a barrier for ML engineers; a higher-level
policy DSL compiled to BPF would improve usability.
Most importantly, our evaluation demonstrates that the mechanism works
but does not yet show that adaptive eBPF policies improve end-to-end
training outcomes. Demonstrating benefit on production workloads at
scale is the key next step.

\paragraph{Conclusion.}
We presented \sysname, a system that transforms NCCL's unsafe native
plugin extension points into verified, composable policy hooks by
embedding a userspace eBPF runtime. The key insight is that NCCL's
CPU-side decision path is latency-tolerant enough to absorb eBPF
overhead. \sysname adds 52--100\ns per decision (under 0.06\% of
collective latency), catches common crash-inducing bugs at load time,
and supports atomic hot-reload ($\sim$7.3\,ms total, 0.774\us on the
hot path) without call loss. The system requires zero NCCL
modifications. Beyond NCCL, the pattern of embedding verified eBPF
execution in latency-tolerant CPU-side decision paths may apply to
other parts of the accelerator software stack.
